# Rules

이 문서는 업데이트가 반복되어도 유지해야 하는 고정 규칙을 기록한다.

일반 기능 추가, 화면 수정, 요구사항 업데이트 과정에서는 이 파일을 수정하지 않는다. 이 규칙을 바꿔야 할 때는 사용자가 명시적으로 `rules.md를 수정하라`고 요청해야 한다.

## 제품 원칙

- 이 시스템은 체육인 창업지원 사업비 집행 절차를 돕는 업무 시스템이다.
- 창업자가 규정을 직접 해석하지 않아도 사전승인, 집행, 검수, 정산 절차를 따라갈 수 있어야 한다.
- 관리자는 기업별 진행 상태, 누락 서류, 위험 항목, 보완 이력을 빠르게 확인할 수 있어야 한다.
- 시스템의 목적은 자동 승인이 아니라 실수 방지와 검토 보조다.

## 권한 규칙

- founder와 admin 모드는 반드시 분리한다.
- founder는 자기 기업과 자기 기업의 지출 신청, 문서, 보완 요청만 볼 수 있어야 한다.
- admin은 전체 기업과 전체 지출 신청을 검토할 수 있다.
- 사업계획서 최종본 업로드와 교체는 admin만 할 수 있다.
- 창업자는 최종 사업계획서를 임의 수정할 수 없다.

## 사업계획서 기준 규칙

- 지출 신청은 반드시 최종 승인된 사업계획서 항목을 기준으로 작성한다.
- 사업계획서 항목에는 비목과 배정 금액이 있어야 한다.
- 지출 유형은 선택한 사업계획서 항목의 비목에 맞는 항목만 표시한다.
- 저장되는 비목은 사용자가 임의 선택한 값이 아니라 선택된 사업계획서 항목의 비목을 기준으로 한다.
- 비목 명칭은 공식 비목명과 일치해야 한다.

## 표준 비목

다음 비목명을 표준으로 사용한다.

- 일반수용비
- 공공요금 및 제세
- 임차료
- 시설장비 유지비
- 시설비
- 일반용역비
- 재료비

## 검토 및 승인 규칙

- 검토 흐름은 `1차 AI 자동 검토 -> 2차 관리자 최종 승인` 구조를 따른다.
- AI 검토 결과는 참고용이며 자동 승인 근거로 사용하지 않는다.
- 최종 승인, 반려, 보완 요청은 관리자가 수행한다.
- 보완 요청에는 창업자가 이해할 수 있는 구체적인 메시지가 있어야 한다.
- 주요 상태 변경과 관리자 판단은 이력으로 남긴다.

## 금액 및 증빙 규칙

- 사업비 신청과 잔액 계산은 공급가액 기준을 우선한다.
- 부가세는 사업비 집행 대상이 아니므로 화면에서 별도로 분리해 보여준다.
- 공급가액 500만원 이상 지출은 비교견적서 필요 여부를 표시한다.
- 공공요금 및 제세는 세금계산서가 없을 수 있으므로 고지서, 청구서, 납부 영수증 등 대체증빙을 허용할 수 있다.
- 대체증빙 인정 여부는 관리자가 최종 판단한다.

## 기술 및 보안 규칙

- 프론트엔드는 HTML5 + ESM JavaScript 구조를 유지한다.
- 인증과 기본 데이터 권한은 Supabase Auth, PostgreSQL, RLS를 사용한다.
- 파일은 Cloudflare R2 private bucket에 저장한다.
- R2 접근은 presigned URL을 통해서만 처리한다.
- Gemini API key, R2 secret key, Supabase service role key는 브라우저에 노출하지 않는다.
- 민감 키는 서버, Supabase Edge Function, 또는 안전한 백엔드 환경에만 둔다.
- 데모 모드는 운영 흐름에 다시 넣지 않는다.
- 업로드 파일의 제목, 원본 파일명, 문서 내용은 한글과 영어를 모두 지원해야 하며 UTF-8 인코딩을 기준으로 처리한다.
- 파일명 저장, 문서 텍스트 추출, AI 분석, 화면 표시 과정에서 한글이 깨지지 않아야 한다.

## 문서 관리 규칙

- 원문 사업 문서와 양식은 `docs/` 폴더에 보관한다.
- 앞으로 변경되는 작업 계획과 요구사항은 `update.md`에 기록한다.
- 바뀌면 안 되는 원칙은 `rules.md`에 기록한다.
- `rules.md`는 사용자의 명시적 요청 없이 수정하지 않는다.
- 코드, 문서, 설정 파일을 수정하기 전에는 반드시 `rules.md`를 먼저 읽고 현재 규칙을 확인한다.
- 수정 작업은 `rules.md`의 고정 규칙과 충돌하지 않아야 한다.
