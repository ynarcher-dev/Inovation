# 사업비 집행 도우미 개발 기획 문서

이 문서는 현재 서비스의 핵심 업무 흐름을 개발 가능한 단위로 정리한 문서다. 고정 원칙은 `rules.md`를 따르며, 이 문서는 구현 계획과 요구사항을 구체화하기 위해 갱신한다.

## 1. 서비스 목적

창업자는 사업비 규정을 직접 해석하지 않아도 시스템의 단계 안내에 따라 예산안을 작성하고, 승인된 예산 범위 안에서 예산 사용 승인을 요청할 수 있어야 한다.

관리자는 사업별 예산 비목을 세팅하고, 기업별 예산안과 예산 사용 요청을 검토하며, 누락 서류와 보완 이력을 확인할 수 있어야 한다.

시스템의 목적은 자동 승인이 아니라 실수 방지와 검토 보조다. AI를 도입하더라도 최종 판단은 관리자가 수행한다.

## 2. 핵심 업무 흐름

### 2.1 사업 및 예산 비목 세팅

1. 관리자가 참가 사업을 생성한다.
2. 관리자가 해당 사업에 적용되는 예산 비목 구조를 정의한다.
3. 관리자가 사업 설명, 안내 사항, 필수 증빙 기준, 내부 메모를 세팅한다.
4. 창업자는 가입 시 참가 사업을 선택하거나, 관리자가 가입 승인 과정에서 사업을 매칭한다.

개발 포인트:

- 사업과 예산 비목은 프로그램 단위로 관리한다.
- 예산 비목은 대분류, 중분류, 소분류 등 사업별로 다른 계층 구조를 허용한다.
- 지출 신청 시 비목은 사용자가 임의 입력하지 않고, 승인된 예산 항목에서 선택하도록 한다.

### 2.2 창업자 가입 및 로그인 승인

1. 창업자가 회원가입을 신청한다.
2. 관리자가 가입 신청 기업을 검토한다.
3. 관리자가 승인하면 창업자는 로그인 후 본인 기업 화면에 접근할 수 있다.
4. 반려된 기업은 서비스 주요 기능에 접근할 수 없다.

개발 포인트:

- 가입 승인 상태와 예산안 승인 상태를 분리한다.
- `companies.approval_status`는 가입/참여기업 승인 상태로만 사용한다.
- 창업자는 본인 기업 데이터만 조회/수정할 수 있어야 한다.

권장 상태:

```text
pending_signup -> approved -> rejected
```

### 2.3 최초 예산안 작성 및 승인

1. 가입 승인을 받은 창업자는 사업에 정의된 예산 비목을 확인한다.
2. 창업자는 각 비목별 구체적인 예산 배정 금액을 입력한다.
3. 창업자는 예산안을 관리자에게 승인 요청한다.
4. 관리자는 예산안을 검토한다.
5. 관리자가 승인하면 해당 금액이 기업의 확정 예산으로 반영된다.
6. 예산안이 승인된 이후에만 창업자는 예산 사용 승인 요청을 할 수 있다.

개발 포인트:

- 기업별 예산 상태는 가입 승인 상태와 분리한다.
- 승인 전 예산안과 승인 완료 예산을 구분해서 저장해야 한다.
- 승인된 예산은 지출 신청의 기준 데이터가 된다.
- 승인 전에는 지출 신청 버튼을 비활성화하고 안내 문구를 보여준다.

권장 상태:

```text
not_submitted -> budget_submitted -> budget_revision_requested -> budget_approved -> budget_rejected
```

### 2.4 예산 변경 요청 및 승인

예산 변경은 최초 예산안 승인 이후에도 발생할 수 있다. 창업자가 관리자에게 예산 변경을 희망하고, 관리자가 승인하면 확정 예산에 할당된 금액이 변경된다.

1. 창업자가 현재 승인된 예산안을 확인한다.
2. 창업자가 변경을 원하는 비목의 금액을 수정하여 예산 변경 요청을 제출한다.
3. 시스템은 기존 승인 예산과 변경 요청안을 동시에 보존한다.
4. 관리자는 변경 전/후 금액과 변경 사유를 비교하여 검토한다.
5. 관리자가 승인하면 변경 요청안이 새로운 확정 예산으로 반영된다.
6. 관리자가 보완 요청하면 창업자는 변경안을 수정해 다시 제출한다.
7. 관리자가 반려하면 기존 승인 예산은 유지된다.

개발 포인트:

- 예산 변경 요청 중에도 기존 승인 예산은 유지되어야 한다.
- 변경 요청이 승인되기 전까지 지출 신청 기준은 기존 승인 예산이다.
- 이미 승인된 예산 사용 요청이나 집행 완료 금액보다 낮게 예산을 줄일 수 없도록 검증해야 한다.
- 변경 요청에는 변경 사유를 필수로 입력하도록 한다.
- 변경 전/후 금액, 증감액, 관리자 의견, 승인/반려 이력을 남긴다.

권장 상태:

```text
change_draft -> change_submitted -> change_revision_requested -> change_approved -> change_rejected
```

권장 데이터 모델:

```text
budget_submissions
- id
- company_id
- type: initial | change
- status
- reason
- submitted_by
- submitted_at
- reviewed_by
- reviewed_at
- review_comment
- created_at

budget_submission_items
- id
- budget_submission_id
- support_program_budget_id
- previous_allocated_amount
- requested_allocated_amount
- approved_allocated_amount

company_budget_allocations
- 현재 승인된 기업별 확정 예산만 저장
```

### 2.5 예산 사용 승인 요청

1. 예산안 승인을 받은 창업자는 승인된 예산 항목 중 하나를 선택한다.
2. 창업자는 예산 사용 목적, 업체명, 공급가액, 부가세, 총액, 예정일, 선금 여부 등을 입력한다.
3. 시스템은 선택한 예산 항목의 비목에 맞는 지출 유형과 필수 서류를 안내한다.
4. 창업자는 필수 첨부서류를 업로드하거나 자동작성 가능한 서류를 작성한다.
5. 창업자는 예산 사용 승인 요청을 제출한다.

개발 포인트:

- 공급가액 기준으로 예산 잔액을 계산한다.
- 부가세는 사업비 집행 대상이 아니므로 별도 표시한다.
- 공급가액 500만원 이상이면 비교견적서 필요 여부를 표시한다.
- 공공요금 및 제세는 대체증빙을 허용할 수 있도록 한다.
- 신청 금액이 승인 예산 잔액을 초과하면 제출을 막는다.

권장 상태:

```text
draft -> pre_approval_submitted -> pre_approval_revision_requested -> pre_approved -> rejected
```

### 2.6 관리자 검토 및 첨부서류 확인

1. 관리자는 예산 사용 승인 요청 목록을 확인한다.
2. 관리자는 신청 정보, 비목, 금액, 잔액, 첨부서류 제출 상태를 확인한다.
3. 관리자는 필요 시 보완 요청, 반려, 승인 중 하나를 선택한다.
4. 보완 요청에는 창업자가 이해할 수 있는 구체적인 메시지를 입력한다.
5. 승인되면 창업자는 오프라인에서 예산을 사용한다.

개발 포인트:

- 관리자 검토 화면은 누락 서류, 위험 경고, 금액/비목 적합성을 한 화면에 보여준다.
- 주요 상태 변경과 관리자 판단은 이력으로 남긴다.
- AI를 도입하는 경우 문서 유형, 금액, 업체명, 날짜, 날인 여부 추출까지만 보조한다.
- AI 결과는 자동 승인 근거로 사용하지 않는다.

### 2.7 승인 후 오프라인 집행

1. 관리자가 예산 사용을 승인한다.
2. 창업자는 오프라인 환경에서 실제 예산을 사용한다.
3. 이후 단계에서 집행 증빙, 검수, 정산 기능을 확장할 수 있다.

추후 확장 상태:

```text
pre_approved -> executing -> execution_submitted -> inspection_submitted -> settlement_submitted -> completed
```

## 5. 데이터 설계 방향

현재 스키마에는 `companies`, `support_programs`, `support_program_budgets`, `company_budget_allocations`, `expense_requests`, `uploaded_files`, `reviews`, `audit_logs` 등의 기본 구조가 있다.

추가 또는 분리 검토가 필요한 구조:

- 가입 승인 상태: `companies.approval_status`
- 기업 예산안 상태: `companies.budget_status` 또는 `budget_submissions.status`
- 예산 변경 요청: `budget_submissions.type = change`
- 예산 변경 항목: `budget_submission_items`
- 확정 예산: `company_budget_allocations`
- 예산 사용 요청: `expense_requests`
- 첨부서류: `uploaded_files`
- 검토 의견: `reviews`
- 상태 변경 이력: `audit_logs`

중요 원칙:

- `company_budget_allocations`에는 현재 승인된 확정 예산만 반영한다.
- 제출 중이거나 검토 중인 예산안은 별도 submission 테이블에 둔다.
- 예산 변경 요청이 반려되어도 기존 확정 예산은 바뀌지 않는다.
- 예산 사용 요청은 항상 현재 확정 예산을 기준으로 검증한다.

## 7. 구현 우선순위

### 1단계: 상태 분리

- 가입 승인 상태와 예산안 승인 상태 분리
- 관리자 대시보드 업무 큐 분리
- 창업자 화면에서 예산안 승인 전 지출 신청 제한

### 2단계: 최초 예산안 승인 흐름

- 창업자 예산안 제출
- 관리자 예산안 승인/반려/보완 요청
- 승인 시 확정 예산 반영

### 3단계: 예산 변경 요청 흐름

- 창업자 예산 변경 요청 작성
- 기존 예산과 변경안 비교
- 관리자 승인 시 확정 예산 갱신
- 반려 시 기존 예산 유지

### 4단계: 예산 사용 승인 요청 고도화

- 승인된 예산 항목 기준 지출 신청
- 잔액 초과 방지
- 필수 서류 체크리스트 연결
- 관리자 검토 화면 개선

### 5단계: 파일 및 AI 검토 보조

- R2 업로드/다운로드 운영 흐름 연결
- 업로드 파일 삭제 UI 연결
- AI 문서 분석 결과 표시
- AI 결과와 관리자 판단 분리

### 6단계: 집행 이후 단계 확장

- 집행 증빙 제출
- 검수조서 작성/업로드
- 정산 내역서 작성 또는 엑셀 업로드
- 최종 제출 패키지 다운로드

## 8. 현재 코드 기준 주의사항

- 현재 `src/api.js`는 실제 Supabase API가 아니라 mock API를 감싸고 있다.
- 현재 mock 흐름에서는 `company.approval_status`가 가입 승인과 예산 승인 의미를 함께 갖는 부분이 있으므로 분리해야 한다.
- `rulesEngine.js`에는 표준 비목, 지출 유형, 필수 서류, 경고 규칙이 일부 구현되어 있다.
- `status.js`에는 사전승인 이후 단계까지 상태 라벨이 정의되어 있으나 실제 화면 흐름은 아직 일부만 연결되어 있다.
