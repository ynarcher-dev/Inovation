# Admin 개발 기획 문서

이 문서는 현재 구현된 founder 화면을 기준으로 admin 화면을 다시 설계하고 개발하기 위한 문서다. 다른 에이전트가 이 문서를 보고 admin 기능을 구현할 수 있도록 메뉴, 화면별 기능, 상태 처리, 개발 우선순위를 정리한다.

고정 규칙은 `rules.md`를 따른다. 특히 다음 원칙은 admin 개발 중 바꾸면 안 된다.

- founder와 admin 모드는 분리한다.
- founder는 본인 기업 데이터만 볼 수 있다.
- admin은 전체 기업, 전체 예산 제출, 전체 지출 신청을 검토할 수 있다.
- 사업계획서 최종본 업로드와 교체는 admin만 할 수 있다.
- AI 검토 결과는 참고용이며 자동 승인 근거로 사용하지 않는다.
- 최종 승인, 반려, 보완 요청은 admin이 수행한다.
- 주요 상태 변경과 관리자 판단은 이력으로 남긴다.

## 1. 현재 Founder 기준 흐름

admin은 현재 founder 화면에서 발생하는 요청을 처리하는 역할로 설계한다.

### 1.1 Founder 대시보드

현재 founder 대시보드는 다음 탭으로 구성되어 있다.

- 지출 현황
- 예산 관리
- 예산 히스토리
- 사업개요 및 첨부파일

founder가 할 수 있는 주요 작업:

- 본인 기업 정보와 참가 사업 확인
- 확정 예산, 사용 금액, 잔액 확인
- 최초 예산안 제출
- 승인된 예산 기준 예산 변경 요청
- 예산 제출/변경 이력 확인
- 승인된 예산 항목을 선택해 지출 신청 생성
- 지출 신청 시 첨부서류 업로드
- 지출 상세 화면에서 제출 내용과 검토 결과 확인
- 사업개요 및 관리자 등록 첨부파일 다운로드

admin은 이 모든 founder 작업의 반대편 처리 화면을 제공해야 한다.

## 2. Admin의 핵심 역할

admin은 운영자가 하루 업무를 처리하는 백오피스다. 단순 조회 페이지보다 “처리해야 할 요청을 빠르게 찾고 판단하는 구조”가 중요하다.

admin의 핵심 업무:

1. 사업 생성 및 사업별 설정 관리
2. 사업별 예산 비목 구조 관리
3. 사업개요 및 첨부파일 관리
4. 창업자 가입 승인/반려
5. 기업별 최초 예산안 검토
6. 기업별 예산 변경 요청 검토
7. 지출 신청 검토
8. 첨부서류 제출 여부 확인
9. 승인/반려/보완 요청 및 검토 이력 관리
10. 기업별 사업계획서 최종본 관리
11. 기업별 내부 메모 및 특이사항 관리

## 3. Admin 메뉴 구조

왼쪽 사이드바는 다음 메뉴를 기준으로 정리한다.

```text
관리자
- 대시보드
- 가입 신청 관리
- 사업 관리
- 기업 관리
- 예산 검토
- 지출 검토
- 규정 및 첨부파일 관리
- 로그아웃
```

기존 메뉴 중 `참가 사업명 관리`와 `참가 사업 관리`가 분리되어 있다면, 가능하면 `사업 관리` 안에서 통합하거나 명확히 역할을 나눈다.

권장 구분:

- `사업 관리`: 사업 생성, 사업명, 사업코드, 활성화, 사업개요, 내부 메모
- `사업 운영 설정`: 예산 비목 구조, 첨부파일, 안내자료, 필수 증빙 기준

단, 1차 개발에서는 메뉴를 많이 늘리기보다 `사업 관리` 한 화면에 탭으로 묶어도 된다.

## 4. Admin 대시보드

대시보드는 admin이 로그인 후 가장 먼저 보는 업무 큐 화면이다.

### 4.1 필수 카드

상단에는 처리해야 하는 건수를 카드로 보여준다.

- 가입 승인 대기
- 최초 예산안 검토 대기
- 예산 변경 검토 대기
- 지출 신청 검토 대기
- 보완/재검토 대기

카드를 클릭하면 해당 목록 위치로 이동하거나 해당 전용 메뉴로 이동한다.

### 4.2 대시보드 하단 목록

대시보드에는 최소한 다음 목록을 제공한다.

- 가입 승인 대기 목록
- 예산안/예산 변경 검토 대기 목록
- 지출 신청 검토 대기 목록
- 최근 반려/보완 요청 이력

### 4.3 대시보드 표시 항목

가입 승인 대기:

- 기업명
- 대표자명
- 참가 사업
- 신청일
- 처리 버튼

예산 검토 대기:

- 기업명
- 참가 사업
- 요청 유형: 최초 예산안 | 예산 변경
- 제출일
- 요청 사유
- 검토 진입 버튼

지출 검토 대기:

- 기업명
- 신청명
- 사업계획서 항목
- 공급가액
- 부가세
- 제출일
- 첨부서류 상태
- 검토 진입 버튼

## 5. 가입 신청 관리

### 5.1 목적

창업자 회원가입 신청을 admin이 검토하고 서비스 접근 권한을 부여한다.

### 5.2 기능

- 가입 신청 기업 목록 조회
- 기업명, 대표자명, 연락처, 사업자등록번호, 참가 사업 확인
- 가입 승인
- 가입 반려
- 반려 사유 입력

### 5.3 상태

```text
pending -> approved -> rejected
```

주의:

- 가입 승인 상태는 `companies.approval_status`로 관리한다.
- 예산 승인 상태와 섞으면 안 된다.
- 가입이 승인되지 않은 founder는 로그인 또는 주요 기능 접근이 제한되어야 한다.

## 6. 사업 관리

### 6.1 목적

admin이 지원사업을 만들고, founder 화면에 표시될 사업 설정을 관리한다.

### 6.2 기능

- 사업 생성
- 사업명 수정
- 사업코드 확인
- 사업 활성화/비활성화
- 사업 설명 작성
- 관리자 내부 메모 작성
- 사업별 첨부파일/안내자료 등록
- 사업별 예산 비목 구조 관리

### 6.3 예산 비목 구조

사업마다 다른 계층 구조를 허용한다.

예:

```text
대분류 > 중분류 > 소분류
```

또는

```text
항목 > 세부항목
```

필수 기능:

- 단계명 수정
- 비목 추가
- 비목 수정
- 비목 삭제
- 정렬
- leaf 항목에 공식 비목명 또는 지출 기준 연결

주의:

- founder의 지출 신청은 승인된 예산 배정 항목을 기준으로 만들어진다.
- 지출 신청 저장 시 `business_plan_item_id` 또는 기업별 예산 배정 id처럼 안정적인 키를 저장해야 한다.
- 비목명 문자열만으로 지출을 연결하면 나중에 비목명이 변경될 때 이력이 깨질 수 있다.

## 7. 기업 관리

### 7.1 기업 목록

기업 목록은 전체 기업 상태를 빠르게 훑는 화면이다.

표시 항목:

- 기업명
- 대표자명
- 참가 사업
- 가입 상태
- 예산 상태
- 총 승인 예산
- 승인 금액
- 검토 중 금액
- 잔액
- 지출 신청 건수
- 상세 관리 버튼

필터:

- 참가 사업
- 가입 상태
- 예산 상태
- 지출 검토 상태

검색:

- 기업명
- 대표자명
- 사업자등록번호

### 7.2 기업 상세

기업 상세는 admin의 핵심 화면이다. founder 대시보드와 같은 정보를 admin 관점으로 확인하고, 검토 액션을 수행한다.

권장 탭:

```text
1. 요약
2. 예산 현황
3. 예산 검토
4. 지출 검토
5. 사업계획서
6. 검토 이력
7. 내부 메모
```

## 8. 기업 상세: 요약 탭

### 8.1 표시 정보

- 기업명
- 대표자명
- 사업자등록번호
- 연락처
- 참가 사업
- 협약 기간
- 가입 상태
- 예산 상태
- 총 승인 예산
- 승인 지출 금액
- 검토 중 지출 금액
- 잔액
- 보완/반려 건수

### 8.2 빠른 액션

- 예산 검토로 이동
- 지출 검토로 이동
- 사업계획서 업로드/교체로 이동
- 내부 메모 작성

## 9. 기업 상세: 예산 현황 탭

### 9.1 목적

현재 확정된 기업별 예산 배정과 지출 사용 현황을 확인한다.

### 9.2 표시 항목

비목 트리 기준으로 다음 값을 보여준다.

- 배정 금액
- 승인된 지출 금액
- 검토 중 지출 금액
- 잔액

계산 기준:

```text
잔액 = 배정 금액 - 승인 금액 - 검토 중 금액
```

주의:

- 검토 중 지출도 잔액에서 미리 차감한다.
- 지출은 가능한 한 예산 배정 id 또는 leaf id 기준으로 연결한다.
- 비목명 문자열만으로 집계하지 않는다.

## 10. 기업 상세: 예산 검토 탭

### 10.1 목적

founder가 제출한 최초 예산안 또는 예산 변경 요청을 admin이 검토한다.

### 10.2 요청 유형

```text
initial: 최초 예산안
change: 예산 변경 요청
```

### 10.3 표시 정보

- 요청 유형
- 제출일
- 제출자
- 변경 사유
- 현재 예산 상태
- 비목별 변경 전 금액
- 비목별 요청 금액
- 증감액
- 총액 변화
- 감액 불가 경고

### 10.4 검토 액션

admin은 다음 중 하나를 선택한다.

- 승인
- 보완 요청
- 반려

보완 요청/반려 시 검토 의견은 필수다.

승인 시:

- `budget_submissions.status`를 승인 상태로 변경한다.
- `budget_submission_items.approved_allocated_amount`를 확정한다.
- `company_budget_allocations`에 확정 예산을 반영한다.
- `companies.budget_status`를 최신 예산 상태로 변경한다.
- 검토 이력을 남긴다.

반려 시:

- 기존 확정 예산은 유지한다.
- 요청안은 반려 상태로 남긴다.
- 검토 의견을 남긴다.

보완 요청 시:

- 기존 확정 예산은 유지한다.
- founder가 다시 작성할 수 있도록 예산 상태를 보완 요청으로 둔다.
- 검토 의견을 남긴다.

### 10.5 상태

최초 예산안:

```text
not_submitted -> budget_submitted -> budget_revision_requested -> budget_approved
                                      -> budget_rejected
```

예산 변경:

```text
change_submitted -> change_revision_requested -> change_approved
                 -> change_rejected
```

### 10.6 중요 정책

- 예산 제출만으로 확정 예산을 바꾸면 안 된다.
- 승인 시점에만 `company_budget_allocations`를 갱신한다.
- 예산 변경 요청이 반려되어도 기존 확정 예산은 유지한다.
- 승인 직전 현재 지출 커밋 금액보다 낮게 감액되는지 다시 검증한다.
- 진행 중인 예산 제출안을 삭제하지 말고 이력으로 남긴다.

## 11. 기업 상세: 지출 검토 탭

### 11.1 목적

founder가 신청한 예산 사용 승인 요청을 admin이 검토한다.

### 11.2 목록 표시 항목

- 신청명
- 사업계획서 항목
- 비목 경로
- 거래처명
- 공급가액
- 부가세
- 총액
- 제출일
- 상태
- 첨부서류 제출률
- 위험 경고 수
- 상세 검토 버튼

### 11.3 지출 상세 검토

지출 상세 검토 화면은 다음 정보를 한 화면에 보여준다.

신청 정보:

- 신청명
- 기업명
- 대표자명
- 사업계획서 항목
- 지출 목적
- 거래처명
- 거래처 사업자등록번호
- 공급가액
- 부가세
- 지출 예정일
- 선금 신청 여부

예산 적합성:

- 해당 항목 배정 금액
- 이미 승인된 금액
- 검토 중 금액
- 이번 신청 금액
- 승인 후 잔액
- 잔액 초과 여부

첨부서류:

- 필수서류 목록
- 제출 여부
- 원본 파일 다운로드
- 기타 증빙서류 목록
- 선금 신청 시 추가 서류

검토 보조:

- 누락 서류
- 500만원 이상 비교견적서 필요 여부
- 부가세 별도 표기
- 공공요금 및 제세 대체증빙 허용 안내
- AI 분석 결과가 있다면 참고 정보로 표시

### 11.4 검토 액션

admin은 다음 중 하나를 선택한다.

- 승인
- 보완 요청
- 반려

승인 시:

- 지출 상태를 승인 상태로 변경한다.
- 승인 금액이 예산 현황에 반영된다.
- 검토 이력을 남긴다.

보완 요청 시:

- 구체적인 보완 메시지를 입력한다.
- founder 상세 화면에 검토 결과로 노출한다.
- 현재 founder UX가 “수정 대신 새 결재” 정책이라면 문구를 `보완 요청`보다 `재신청 요청`에 가깝게 조정할지 논의가 필요하다.

반려 시:

- 반려 사유를 필수 입력한다.
- 검토 이력을 남긴다.

## 12. 사업계획서 관리

### 12.1 현재 주의점

`rules.md` 기준으로 최종 사업계획서 업로드와 교체는 admin만 가능하다.

따라서 admin 개발 시 다음 중 하나를 선택해야 한다.

정책 A:

- founder가 올리는 파일은 `검토용 사업계획서 제출본`이다.
- admin이 검토 후 `최종 사업계획서`로 확정한다.

정책 B:

- founder는 사업계획서를 올릴 수 없다.
- admin만 기업 상세에서 최종 사업계획서를 업로드/교체한다.

현재 founder 화면에 파일 첨부/수정 UI가 있다면, admin 개발 전에 정책 A/B 중 하나를 정해야 한다.

### 12.2 Admin 사업계획서 기능

필수 기능:

- 기업별 최종 사업계획서 업로드
- 기존 파일 다운로드
- 파일 교체
- 최종 수정일 표시
- 변경 이력 표시

권장 표시:

- 현재 최종본 파일명
- 업로드일
- 업로드한 관리자
- 이전 버전 목록

## 13. 규정 및 첨부파일 관리

### 13.1 목적

founder의 `사업개요 및 첨부파일` 탭에 표시될 사업 설명과 첨부파일을 admin이 관리한다.

### 13.2 기능

- 사업별 설명 작성
- 안내자료 제목 입력
- 첨부파일 업로드
- 첨부파일 다운로드 확인
- 첨부파일 삭제
- 노출 순서 관리

주의:

- 파일은 실제 운영에서 R2 private bucket에 저장한다.
- 브라우저에는 R2 secret이나 service role key가 노출되면 안 된다.
- 다운로드는 presigned URL 또는 안전한 Edge Function을 사용한다.

## 14. 내부 메모 및 검토 이력

### 14.1 내부 메모

admin 전용 메모는 founder에게 노출하지 않는다.

필수 기능:

- 기업별 내부 메모 작성
- 저장
- 마지막 수정일 표시

### 14.2 검토 이력

예산 검토와 지출 검토는 모두 이력으로 남겨야 한다.

표시 항목:

- 대상 유형: 예산안 | 예산 변경 | 지출 신청 | 사업계획서
- 처리 결과: 승인 | 보완 요청 | 반려
- 관리자
- 처리일
- 검토 의견

권장:

- `reviews`는 화면 호환용으로 유지할 수 있다.
- 장기적으로는 `audit_logs` 또는 전용 검토 이력 테이블에 상태 변경을 남긴다.

## 15. 정책 결정 필요 항목

내일 부서와 논의할 때 아래 질문을 확인한다.

1. founder가 업로드하는 사업계획서는 최종본인가, 검토 요청본인가?
2. 최종 사업계획서는 admin만 올리는 방식으로 확정할 것인가?
3. 필수서류가 모두 있어야 지출 신청이 접수되는가?
4. 누락 서류가 있어도 접수 후 admin이 보완 요청하는 방식인가?
5. 보완 요청을 받은 지출 신청은 같은 건에서 수정하는가, 새 신청으로 다시 올리는가?
6. 예산 변경 검토 중 기존 확정 예산으로 새 지출 신청을 허용할 것인가?
7. admin이 기업 예산을 직접 조정할 수 있어야 하는가?
8. admin 직접 조정이 필요하다면 사유 입력과 이력 기록을 필수로 할 것인가?
9. AI 문서 분석을 이번 단계에 넣을 것인가, 추후 단계로 미룰 것인가?
10. 승인 후 오프라인 집행 이후의 검수/정산 단계까지 이번 범위에 포함할 것인가?

## 17. 구현 시 주의사항

### 17.1 현재 mock API 기반

현재 `src/api.js`는 실제 Supabase API가 아니라 mock API를 감싼다. admin도 우선 mock 데이터 기준으로 개발한다.

### 17.2 실제 Supabase 연동 전 확인

기존 `submit_founder_budget_allocations` RPC는 제출 즉시 확정 예산을 바꾸는 구조일 수 있다. 새 기획에서는 예산 제출과 확정 예산 반영을 분리해야 하므로 실제 DB 연동 전 RPC를 반드시 재설계한다.

### 17.3 첨부 파일 다운로드

mock 환경에서도 업로드 파일 레코드에 `link_url`이 있어야 다운로드가 정상 동작한다. 사업계획서/안내자료/지출 증빙 모두 같은 파일 저장 흐름을 쓰는 것이 좋다.

### 17.4 필수서류 정책

현재 founder UX가 “신청 접수 후 수정 불가”라면 제출 전 필수서류 검증이 필요하다. 반대로 admin 보완 요청을 살릴 거면 founder 상세에서 보완 업로드/재제출 동선이 필요하다.

### 17.5 Admin 직접 수정

admin이 확정 예산을 직접 수정하는 기능은 편리하지만, founder 예산 변경 요청 흐름과 충돌할 수 있다. 유지한다면 다음을 필수로 한다.

- 직접 조정 사유 입력
- 변경 전/후 금액 기록
- 관리자 기록
- founder 화면의 예산 히스토리에 표시

## 18. 개발 완료 기준

1차 admin 프로토타입 완료 기준:

- admin 대시보드에서 처리 대기 건을 볼 수 있다.
- 기업 상세에서 founder가 보는 예산/지출 정보를 admin 관점으로 볼 수 있다.
- 최초 예산안과 예산 변경 요청을 승인/보완/반려할 수 있다.
- 지출 신청을 승인/보완/반려할 수 있다.
- 첨부서류 제출 여부와 파일 다운로드를 확인할 수 있다.
- 검토 의견이 founder 상세 화면 또는 히스토리에 남는다.
- 사업별 설명과 첨부파일이 founder의 사업개요 탭에 반영된다.

이번 단계의 목표는 운영 확정본이 아니라, 내일 부서와 “이 업무 흐름이 맞는지” 확인할 수 있는 clickable admin prototype이다.
